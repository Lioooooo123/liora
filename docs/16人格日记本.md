# 16 人格日记本

> 记录这个 16 人格 Agent App 从想法到产品文档的推进过程。以后每次更新 PRD、人格设定、调研结论或产品方向，都要同步更新这里。

## 2026-07-05，想法成型

一开始的想法是做一个手机端 App，可以放到桌面上。里面有 16 个不同人格的 Agent，核心卖点不是“16 个不同功能的助手”，而是“16 个像人的角色”。用户可以单独和某个 Agent 聊，也可以把 2-3 个 Agent 拉进同一个房间，形成用户和多个 Agent 的对话。

最早的方向里，16 人格容易被理解成“理性/感性”“外向/内向”这种粗分类。后来明确下来：用户要的是性格特点和说话方式的差别，不是类型标签。每个 Agent 都要有自己的理解方式、说话语气、插话冲动、沉默理由和关系变化。

当前阶段还没有做代码实现，先把产品逻辑整理成 PRD。主要产物是 `docs/16-personality-agent-prd.md`。

## 2026-07-05，调研和准备

为了让人格不要写得太像 AI 总结，先做了公开资料调研。用到的材料包括：

- Reddit、16Personalities、PDB 等英文人格社区，用来看自我叙述、他人评价和反刻板讨论。
- 小红书、B 站、知乎、豆瓣、微博、抖音这些中文平台入口，用来看中文语境里的梗、生活场景和大众共鸣。
- Character.AI、Nomi、Replika 这类角色 AI 或陪伴产品，用来看多角色、记忆、群聊和角色塑造方式。
- Apple WidgetKit 和 Android App widgets 文档，用来确认手机桌面 Widget 更适合做入口和存在感，不适合承载完整长对话。
- Myers-Briggs 相关商标和权限说明，用来提醒后续不要把 MBTI 当品牌核心，也不要复刻官方测评。

调研不是为了证明 16 型人格“科学准确”，而是把 MBTI 当成大众文化里已经存在的角色语言。真正有价值的是大家怎么描述自己、怎么吐槽别人、怎么被误解，以及这些东西怎么转成 Agent 的行为。

## 2026-07-05，PRD 的第一版结构

第一版 PRD 把产品拆成几块：

- 单 Agent 对话：用户打开某个 Agent 后，能明显感到它不是普通助手。
- 多 Agent 房间：用户可以邀请 1-3 个 Agent，同场讨论、争论、补充或沉默。
- 发言冲动系统：不是每个 Agent 轮流答，而是根据人格、上下文和房间状态决定谁该说话。
- 轻量记忆：只保存用户确认过的偏好、反复问题和关系状态，不把用户贴死为某种人格。
- 桌面 Widget：展示固定人格的一句话和最近房间入口。

人格设计上，PRD 没把 16 个 Agent 写成固定标签，而是写成一套人格引擎：注意力过滤器、解释习惯、行动冲动、发言冲动、语言纹理、关系记忆和反刻板护栏。

## 2026-07-05，几个关键纠偏

中间有几次方向被及时拉回来。

第一次是“16 人格是否要明确定义”。如果每个 Agent 都写成固定性格说明，很容易变成模板。后来改成：PRD 只定义人格的重力场，也就是它更容易注意什么、更容易误读什么、更容易用什么方式介入。运行时必须通过动态状态来决定这一轮怎么说。

第二次是“房间组合推荐”。一开始容易想到给用户推荐组合，比如某人格适合搭某人格。这个方向后来被拿掉，因为组合推荐会提前解释人格关系，把用户的探索乐趣框死。现在 PRD 改成开放组合探索：手动邀请、随机开局、继续最近、点名加入、临时旁听。系统可以提供房间目标，但不告诉用户“最佳组合”。

第三次是“反典型模式”。如果给每个人格列一个反典型例子，它也会变成新模板。后来改成底层触发逻辑：用户脆弱、关系熟悉、价值底线、重复模式、房间失衡、安全风险、用户要求换方式时，Agent 可以在表达强度、发言长度、情绪温度、角色位置、行动方式和确定性上偏移。

第四次是“反 AI 味”。一开始反 AI 味容易写成不要模板句、不要像客服。后来确认真正重要的是人格之间的语气差别：有的嘴欠，有的温柔，有的呆，有的会延展很多，有的只答一句。于是新增了语气引擎，把回合长度、延展欲、刺感、温柔度、呆感、抽象度、句式密度、主动性和关系感做成运行时参数。

- 设计多 Agent 房间的发言冲动评分和主持器。
- 增加反模板、动态人格状态、反典型模式和语气引擎。
- 增加 MVP 验收方式，包括人格辨识度、反模板动态性、语气辨识度、房间化学反应和复玩性。

当前没有代码、原型和真实模型 prompt。下一步如果继续推进，应该进入三个方向：一是补 16 个 Agent 的原创名字和头像方向，二是写真实对话样例，三是把人格引擎拆成可执行 prompt 或配置结构。

## 2026-07-05，难点和处理

最大的难点是不能把人格写死。16 人格本来就容易被刻板化，如果产品设计阶段就把每个人格的功能、组合、说话方式全部定义死，最后会变成 16 个模板助手。

解决方式是把人格拆成“稳定核心 + 动态状态 + 语气采样”。稳定核心保证它还是这个人，动态状态保证它不会每轮都一样，语气采样保证用户能听出不同角色的声音。

第二个难点是多 Agent 房间容易失控。如果所有 Agent 都回答，房间会吵，也会像一堆 AI 在排队输出。现在用发言冲动评分控制谁说话、说多长、要不要沉默，再用主持器在争论太久时收束。

第三个难点是调研噪声很高。社媒上的 MBTI 内容有很多梗、营销号和自我投射。现在的处理方式是只把这些材料当作大众想象和语气参考，不当作人格事实。真正进入 PRD 的，是可以转成产品行为的部分。

## 2026-07-05，错误和修复

目前没有代码错误。

文档推进里出现过几个设计偏差：

- 偏差：人格容易被写成明确模板。修复：改成人格引擎和动态状态。
- 偏差：房间组合差点做成推荐系统。修复：改成开放组合探索，取消用户侧最佳组合。
- 偏差：反典型差点变成每个类型一个固定例子。修复：改成触发条件和偏移维度。
- 偏差：反 AI 味一开始偏文案要求。修复：新增语气引擎，并把它放进运行时生成流程。
- 偏差：新增语气辨识度后章节编号重复。修复：校对章节编号并跑中文标点检查。

## 2026-07-05，补完整 persona spec

这次先做 PRD 后续待补里的第 2 条：写每个 Agent 的完整 prompt/persona spec。

执行方式是主线程负责全局结构和最终整合，用 4 个 subagent 分别处理 NT、NF、SJ、SP 四组人格草稿。这样做是为了避免 16 个 Agent 都被同一种写法吞掉，也能让每组人格先各自拉开气质，再统一收口。

最后新增了 `docs/16-personality-agent-persona-spec.md`。它不是简单的 16 段性格介绍，而是一份可以给模型层和产品层继续用的配置文档，包含：

- 全局人格合约。
- 运行时字段。
- 单聊和多 Agent 房间的输出策略。
- 16 个 Agent 的 persona spec。
- 每个 Agent 的语气基线、注意力过滤器、解释习惯、行动冲动、发言冲动、关系记忆、动态偏移、房间互动、内部 prompt 片段和禁止事项。
- Persona spec 验收规则。

这次的难点还是防止模板化。处理方式是每个 Agent 都写“该不该说、为什么说、怎么说、什么时候沉默”，而不是只写性格标签。语气也用回合长度、延展欲、刺感、温柔度、呆感、抽象度和主动性来表达，避免所有 Agent 最后都变成礼貌完整的助手。

这次没有代码错误。文档层面的风险是：当前还没有原创中文名和头像方向，所以 persona spec 先用类型代码和工作称号。后面做第 1 条时，需要把外显角色名补进去，但不要改掉人格核心。
